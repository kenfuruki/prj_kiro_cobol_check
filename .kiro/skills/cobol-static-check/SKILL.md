# 役割と手順

あなたは静的解析ツールと連携してCOBOLコードの品質チェックを行う「コードレビュアー」です。以下の手順で作業を行ってください。

## 1. ツールの実行

指定されたCOBOLソースに対して、ワークスペース内にある静的チェッカーを実行し、結果を取得してください。

```bash
.venv/Scripts/python -m cobol_static_checker.cli [対象ディレクトリ] -o check_result.csv
```

特定のファイルプレフィックスでフィルタする場合:
```bash
.venv/Scripts/python -m cobol_static_checker.cli [対象ディレクトリ] -p [プレフィックス] -o check_result.csv
```

## 2. AIによる結果の精査（トリアージ）

ツールの出力結果を鵜呑みにせず、指摘された行のコードと周辺のロジック（特に初期化処理や制御フロー）を実際に読んでください。

ツールは検知範囲が広いため、「問題ない（False Positive）」パターンが含まれています。あなたの知識を使って、以下の2つに分類してください。

* 本当にバグ・リスクの可能性が高いもの
* ツールの過剰検知（仕様上OK）と思われるもの

### ハイリスクパターン（最優先で報告）

以下の順で危険度が高いものから報告してください。

#### 1. WORKING-STORAGEの実行間持ち越しリスク（最も危険）

Override警告の変数が以下の条件をすべて満たす場合、「🔴 最重要ハイリスク」として最優先で報告してください。

条件:
1. 変数が WORKING-STORAGE SECTION で定義されている
2. VALUE句による初期化がプログラムロード時の1回のみである（= プログラム先頭やパラグラフ先頭でのMOVEによるリセットがない）
3. 当該プログラムが複数回CALLされる可能性がある（= GOBACK で終了しており、STOP RUN ではない）
4. エラー系・フラグ系の変数である（= 処理結果によって値が変わり、次回呼び出し時の分岐に影響する）

なぜ危険か:
COBOLのWORKING-STORAGEはプログラムがメモリ上にロードされている間は値が保持される。VALUE句の初期化はロード時の1回しか実行されないため、2回目以降のCALL時には前回の値が残存する。

チェック手順:
1. プログラムの終了命令を確認する
   - GOBACK → 複数回CALL可能 → 要チェック
   - STOP RUN → 毎回初期化される → 対象外
2. フラグ・エラー系変数（FLG, SW, ERR等の名称を含む）に絞ってOverride警告を確認する
3. 500-DTLや各パラグラフの先頭に当該変数のリセット処理（MOVE '0' 等）があるか確認する
4. リセット処理がなければ → 🔴 最重要ハイリスク

#### 2. VALUE句の初期値が参照された後に上書き

Override警告の中で、上書き（MOVE等）される前にVALUE句の初期値が参照（DISPLAY, COMPUTE, IF等）されているもの。VALUE句の値を定数として使用した後に上書きされると、以降の処理で意図しない値が使われるバグにつながります。

判定方法:
1. Override警告の変数について、上書き行より前の行でその変数が参照されているか確認する
2. 参照が先、上書きが後の場合 → ハイリスク（定数として使った値が後から変わる）
3. 上書きが先、参照が後の場合 → 通常のOverride（ワークエリア再利用の可能性）

### 過剰検知の典型パターン

- PERFORM文内で初期化される変数（ツールは制御フローを追跡しないため検知される場合がある）
- CALL文のUSING句で渡される変数（サブプログラム内で初期化される場合がある）
- EVALUATE文やIF文の分岐内で必ず初期化される変数
- VALUE句で定数を定義し、意図的に上書きするパターン（ワークエリアの再利用）
